# Javi - Epic Breakdown

## Overview

Декомпозиция MVP Javi (доставка день-в-день своим курьером с расчётным ETA) из PRD, UX и архитектуры в реализуемые истории.

> **Принцип нарезки (заказчик):** истории режем **вертикально по INVEST** — каждая история это тонкий сквозной срез (UI → логика → данные → прод), который **виден в проде** и объясняется **бизнес-сутью в одном предложении**. Технические энейблеры (init, провайдеры) включаются в первый видимый срез («walking skeleton»), а не выносятся в невидимые горизонтальные истории.

## Requirements Inventory

### Functional Requirements

FR-1: Магазин входит под ролью «магазин» и видит только свои доставки.
FR-2: Магазин задаёт origin (адрес магазина) в профиле; геокодируется.
FR-3: Магазин/курьер быстро создаёт доставку: полный адрес + телефон получателя (+ описание опц.).
FR-4: Телефон валидируется/нормализуется к E.164 (RS); невалидный — блок; немобильный/иностранный — предупреждение без блока.
FR-5: Адрес получателя геокодируется; при неудаче — уточнение/точка на карте или откат на ручной ETA.
FR-6: (later) Внешняя система магазина создаёт доставку через API (в MVP — задел модели данных).
FR-7: Магазин/курьер отмечает «Доставка началась» (триггер расчёта ETA).
FR-8: ETA = время старта + время в пути origin→получатель (Routes API, трафик), Europe/Belgrade.
FR-9: При сбое маршрута/геокода — fallback на ручной ETA; поток не блокируется.
FR-10: ETA показывается как «ориентировочно к HH:MM» (верхняя граница).
FR-11: При старте — сообщение получателю «в пути, к {время}» + ссылка на страницу статуса.
FR-12: Viber-first; при недоставке Viber — авто-fallback на SMS (нативно Infobip).
FR-13: Идемпотентность отправки (logical_message_id), без дублей.
FR-14: Фиксация по сообщению канала/статуса (queued/sent/delivered/read/failed)/времени; read где поддерживается.
FR-15: Шаблоны сообщений на сербском (латиница), строго сервисные.
FR-16: Окно рассылки 08:00–22:00 (Europe/Belgrade); вне окна — откладывание.
FR-17: Публичная страница статуса по непредсказуемому токену, без приложения/логина.
FR-18: Страница показывает магазин, статус «в пути», ETA, город; без телефона/полного адреса.
FR-19: Блок оценки 1–5 на странице после доставки.
FR-20: Rate limit страницы; срок жизни ссылки.
FR-21: Через ETA+30 мин — активный запрос оценки сообщением (с учётом окна и отписки).
FR-22: Получатель ставит оценку 1–5 одним касанием, без логина; без дублей.
FR-23: Отписка (STOP/ссылка); после — не-критичные сообщения не шлются; магазин видит «отписан».
FR-24: Магазин видит по доставке статус уведомления (доставлено/прочитано/не доставлено), ETA, оценку.
FR-25: При недоставке — исправить номер и переотправить (явным подтверждаемым действием).
FR-26: (опц.) Ручная отметка «Доставлено»; система от неё не зависит.

### NonFunctional Requirements

NFR-1: Производительность — расчёт ETA + отправка < 5 секунд после «доставка началась».
NFR-2: Надёжность сообщений — фиксация статусов; не «доставлено» без подтверждения провайдера.
NFR-3: Приватность/ПДн (Сербия) — минимизация данных на публичной странице, непредсказуемый токен, согласие через договор магазина.
NFR-4: Локализация — сербский (латиница) по умолчанию.
NFR-5: Мобайл-first — вид магазина/курьера рассчитан на телефон «на бегу».
NFR-6: Безопасность — изоляция арендаторов, rate limit публичной страницы, OIDC/подпись для тасков/вебхуков.

### Additional Requirements

(из Архитектуры — greenfield Django 6 на Cloud Run)
- AR-1: Инициализация проекта (uv + Django 6 + apps accounts/deliveries/notifications/integrations/tracking/tasks/common); Dockerfile (gunicorn+whitenoise) вместо nginx-образа Этапа 0; деплой по существующему CI/CD (тег → Cloud Run).
- AR-2: Cloud SQL for PostgreSQL; подключение из Cloud Run через connector.
- AR-3: Интеграции за интерфейсами (`integrations/`): Google Maps (Routes API + Geocoding), Infobip (Viber+SMS, receipts, opt-out blocklist).
- AR-4: Async через Cloud Tasks (`scheduleTime`) → защищённые OIDC HTTP-эндпоинты; один механизм на ETA+30 и окно рассылки.
- AR-5: Secret Manager для ключей (Google Maps, Infobip, SECRET_KEY, БД); env-config (12-factor).
- AR-6: Кэш результатов геокодинга (снижение стоимости Maps).
- AR-7: Входящие вебхуки Infobip (delivery/seen receipts + opt-out) → обновление Notification/OptOut.
- AR-8: Модель данных: Shop, Delivery, Notification, TrackingToken, Rating, OptOut.

### UX Design Requirements

UX-DR1: Расщепление тем — светлая утилитарная тема кабинета («Clean») + брендовая (градиент-шапка) публичная страница.
UX-DR2: Компонент delivery_card (имя+адрес+ETA+статус-чип; в «Spremno» — кнопка старта прямо в списке).
UX-DR3: Компонент status_chip (семантика: U dostavi/Pročitano/Isporučeno/Nije dostavljeno — цвет + текст/иконка).
UX-DR4: primary_button и постоянная new_delivery_button (на всю ширину, низ экрана).
UX-DR5: rating_stars (тап = отправка, без кнопки; «Hvala!» после).
UX-DR6: status_stepper (Primljeno · U dostavi · Isporučeno) на публичной странице; крупный ETA — главный элемент.
UX-DR7: Состояния — пусто / загрузка (спиннер на кнопке) / ошибка геокода / недоставлено / отписан / оценка отправлена / вне окна.
UX-DR8: Оптимистичный старт с undo-тостом перед фиксацией отправки.
UX-DR9: Доступность — тач-цели ≥48px, статус не только цветом, aria-label для звёзд, контраст AA, чтение статуса без JS.
UX-DR10: Микрокопирайт на сербском (латиница) по шаблонам из EXPERIENCE.md.

## Epic 1: Магазин ведёт свои доставки

Магазин может войти, задать адрес магазина и завести доставку дня — и видеть её в проде.

### Story 1.1: Вход магазина и кабинет доставок на проде (walking skeleton)

**Бизнес-суть:** магазин входит и видит свой (пустой) экран доставок на `javi.serbito.rs`.

As a владелец магазина,
I want войти в Javi и увидеть свой экран доставок,
So that у меня есть рабочее место для доставок, доступное в проде.

**Acceptance Criteria:**

**Given** инициализированный Django-проект (uv + Django 6 + apps), задеплоенный на Cloud Run по существующему CI/CD, с моделью `Shop` (привязана к пользователю)
**When** зарегистрированный магазин входит по email+паролю на проде
**Then** он видит мобайл-first экран «Доставки» (пока пустой) со светлой темой кабинета и кнопкой «＋ Nova dostava»
**And** чужие магазины не видят данные друг друга (изоляция по `Shop`)
**And** неавторизованный редиректится на вход _(FR-1, AR-1/2/5/8, UX-DR1, NFR-5)_

### Story 1.2: Магазин задаёт адрес своего магазина (origin)

**Бизнес-суть:** магазин один раз сохраняет свой адрес — точку отсчёта для будущих расчётов времени.

As a владелец магазина,
I want указать адрес моего магазина в профиле,
So that система знает, откуда стартует доставка для расчёта ETA.

**Acceptance Criteria:**

**Given** авторизованный магазин в профиле
**When** он вводит и сохраняет адрес магазина
**Then** адрес геокодируется (Google Geocoding) и сохраняется с координатами как `origin`
**And** при неудачном геокоде показывается подсказка уточнить адрес
**And** заданный origin виден и редактируется позже _(FR-2, AR-3/6)_

### Story 1.3: Магазин заводит доставку и видит её в списке

**Бизнес-суть:** магазин добавляет доставку дня (адрес + телефон получателя), и она появляется в списке «готово к старту».

As a владелец магазина/курьер,
I want быстро завести доставку с адресом и телефоном получателя,
So that она готова к отправке и видна в списке дня.

**Acceptance Criteria:**

**Given** магазин с заданным origin
**When** он заполняет форму «Nova dostava»: имя, телефон, полный адрес получателя (+ описание опц.)
**Then** телефон нормализуется к E.164 (+381); невалидный — блок с примером, немобильный/иностранный — предупреждение без блока
**And** адрес получателя геокодируется (при неудаче — уточнить/точка на карте)
**And** доставка появляется в списке в группе «Spremno» карточкой (имя + адрес) _(FR-3, FR-4, FR-5, UX-DR2/4)_

## Epic 2: Получатель узнаёт, что заказ в пути и когда приедет

По «Доставка началась» получатель получает сообщение с расчётным временем + ссылку; магазин видит статусы и чинит сбои.

### Story 2.1: «Доставка началась» → получатель получает сообщение с ETA

**Бизнес-суть:** магазин жмёт «началась» — покупатель получает сообщение, что заказ выехал и ориентировочно к какому времени будет.

As a владелец магазина/курьер,
I want отметить «Доставка началась» и чтобы покупатель сразу узнал время прибытия,
So that покупатель готов принять, а я не отвечаю на «где заказ?».

**Acceptance Criteria:**

**Given** доставка в статусе «Spremno» с геокодированным адресом
**When** магазин/курьер жмёт «Dostava je počela» (с undo-тостом перед фиксацией)
**Then** система рассчитывает ETA = сейчас + время в пути origin→получатель (Routes API, трафик) и показывает «ориентировочно к HH:MM»
**And** получателю уходит сообщение `Vaša porudžbina iz {prodavnica} je u dostavi. Stiže okvirno do {vreme}. {link}` (на сербском, латиница), идемпотентно (logical_message_id)
**And** при сбое маршрута/геокода — fallback на ручной ввод ETA, поток не рвётся
**And** карточка переезжает в группу «U dostavi»; всё < 5 c _(FR-7,8,9,10,11,13,15, NFR-1, UX-DR8)_

### Story 2.2: Публичная страница статуса по ссылке

**Бизнес-суть:** покупатель открывает ссылку и видит магазин, статус «в пути» и время прибытия — без приложения и логина.

As a получатель,
I want открыть ссылку из сообщения и увидеть статус и время,
So that я понимаю, когда будет курьер, и спокоен.

**Acceptance Criteria:**

**Given** отправленное уведомление с трекинг-ссылкой
**When** получатель открывает `/t/<token>` в браузере
**Then** он видит брендовую страницу: магазин, степпер (Primljeno · U dostavi · Isporučeno), крупный ETA «do HH:MM», город назначения
**And** не показываются телефон и полный адрес; токен непредсказуем; есть rate limit и срок жизни ссылки
**And** страница читается без JS _(FR-17,18,20, NFR-3/6, UX-DR6/9)_

### Story 2.3: Viber-first с авто-fallback на SMS

**Бизнес-суть:** сообщение идёт в Viber, а если не дошло — автоматически в SMS, без дублей.

As a получатель,
I want получать уведомление в Viber (а если недоступен — в SMS),
So that сообщение точно дойдёт удобным каналом.

**Acceptance Criteria:**

**Given** интеграция Infobip (Viber Business + SMS)
**When** отправляется уведомление
**Then** первая попытка — Viber; при недоставке (fail/timeout) — авто-fallback на SMS тем же текстом и ссылкой
**And** одно событие не дублируется по каналам (идемпотентность)
**And** если Viber Business ещё не подключён — конфиг-флаг позволяет стартовать на SMS _(FR-12, AR-3)_

### Story 2.4: Магазин видит статус уведомления и чинит сбой

**Бизнес-суть:** магазин видит, что покупатель получил/прочитал сообщение, а при сбое исправляет номер и шлёт повторно.

As a владелец магазина,
I want видеть, дошло ли и прочитано ли уведомление, и переотправить при сбое,
So that ни один покупатель не остаётся без уведомления.

**Acceptance Criteria:**

**Given** отправленные уведомления и вебхуки Infobip (delivery/seen)
**When** провайдер присылает статус
**Then** у доставки в кабинете отображается чип: `Pročitano` / `Isporučeno (delivered)` / `Nije dostavljeno` (цвет + текст)
**And** при `Nije dostavljeno` магазин может исправить телефон и нажать «Pošalji ponovo» (явное подтверждаемое действие, идемпотентность сохраняется)
**And** (опц.) магазин может вручную отметить «Доставлено» — система от этого не зависит _(FR-14,24,25,26, AR-7, UX-DR3/7, NFR-2)_

## Epic 3: Оценка доставки

После доставки получатель оценивает её 1–5; магазин видит оценку.

### Story 3.1: Запрос оценки через ETA+30 и оценка 1–5

**Бизнес-суть:** через полчаса после ожидаемого прибытия покупатель получает запрос и ставит оценку доставке, а магазин видит звёзды.

As a получатель,
I want оценить доставку после получения,
So that магазин понимает качество, а я влияю на сервис.

**Acceptance Criteria:**

**Given** доставка с расчётным ETA
**When** наступает ETA+30 минут (запланировано через Cloud Tasks `scheduleTime`)
**Then** получателю уходит `Kako je prošla dostava iz {prodavnica}? {link}`; если время вне окна 08:00–22:00 — отправка сдвигается на ближайшее открытие окна
**And** на странице статуса получатель ставит оценку 1–5 одним касанием (без логина), затем видит «Hvala!»; повтор не плодит дубли
**And** магазин видит оценку доставки в кабинете _(FR-19,21,22,24-rating,16, AR-4, UX-DR5)_

### Story 3.2: Отписка получателя

**Бизнес-суть:** покупатель может отписаться и больше не получать не-срочные сообщения.

As a получатель,
I want отписаться от сообщений,
So that меня не беспокоят, если я не хочу.

**Acceptance Criteria:**

**Given** полученное сообщение со способом отписки (STOP в SMS / ссылка)
**When** получатель отписывается
**Then** его номер попадает в блоклист (зеркалится с Infobip); не-критичные сообщения (в т.ч. запрос оценки) на него больше не шлются
**And** магазин видит у доставки статус «primalac otkazao obaveštenja» _(FR-23)_
