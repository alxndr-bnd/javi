## 0. Document Purpose

Строится на брифе (`../../briefs/brief-javi-2026-06-01/brief.md`) и каталоге use cases (`../../use-cases-javi.md`) — не дублирует их, а специфицирует MVP. Технические решения (провайдеры, модель данных) — в `addendum.md`, не здесь.

## 1. Vision
Javi возвращает малым магазинам в Сербии клиентский опыт доставки уровня Wolt/Glovo — для их **собственных** курьеров и **без комиссии 30%**. Beachhead — пиццерии/кафе со своей доставкой день-в-день: еда остывает, клиент звонит «где заказ?», передача затягивается.

Магазин отмечает «доставка началась» — Javi считает время прибытия по маршруту и шлёт получателю в Viber/SMS «в пути, прибудет ~15:30» + ссылку на трекинг без приложения; после — активно просит оценку 1–5. Получатель готов к приёму (горячая еда, быстрая передача, меньше звонков), магазин выглядит профессионально и видит качество доставок в цифрах.

Стратегически Javi — анти-агрегатор: магазин сохраняет свою доставку, маржу и прямой контакт с клиентом, получая UX больших платформ.

## 2. Target User

### 2.1 Jobs To Be Done
- **Магазин (Милан):** «когда мой курьер везёт заказ — пусть клиент сам видит, что доставка началась и когда приедет, чтобы был готов принять, не звонил “где?” и остался доволен» — не нанимая агрегатор и не печатая ничего вручную.
- **Получатель (Ана):** «хочу понимать, выехал ли курьер и через сколько будет, чтобы не сидеть в неизвестности и быть на месте».

### 2.2 Key User Journeys

- **UJ-1. Милан отправляет заказ в доставку (день-в-день).**
  - **Персона + контекст:** Милан, владелец маленькой пиццерии в Нови-Саде, один курьер на мопеде. Заказы и адреса ведёт в **своей** системе — не в Javi.
  - **Entry state:** настройка пройдена (в профиле Javi задан адрес магазина = origin). Доставка (адрес + телефон получателя) попала в Javi одним из двух способов (**гибрид**): минимальный быстрый ввод/вставка в Javi (MVP) **или** передача из системы магазина через API (later). Милан/курьер авторизованы в простом мобильном виде Javi.
  - **Path:** пицца готова → курьер берёт заказ → Милан (или курьер) на телефоне открывает в Javi список сегодняшних доставок → у нужной жмёт **«Доставка началась»** (если доставки ещё нет — быстро добавляет адрес+телефон).
  - **Climax:** Javi рассчитывает ETA по маршруту origin → адрес получателя и отправляет Ане в Viber/SMS «в пути, ориентировочно к {время}» + ссылку на страницу статуса. Милан видит, что уведомление ушло.
  - **Resolution:** Милан вернулся к работе. В списке у заказа видит статусы: уведомление **доставлено/прочитано**; если Ана подтвердила — **получено**; если оценила — **оценка 1–5**.
  - **Кто жмёт «началась»:** Милан или курьер — оба могут; по умолчанию полагаемся на Милана (у него заказы). _(решено)_

  > **РЕШЕНО (гибрид):** адрес+телефон попадают в Javi либо быстрым ручным вводом в Javi (MVP, разблокирует мелкие магазины и валидацию), либо через API из системы магазина (later). Полное ведение заказов (каталог, оплата) — НЕ часть Javi; Javi хранит только доставку. Двойной ввод для Милана в MVP — осознанная плата за низкий барьер входа; API его убирает позже.

- **UJ-2. Ана отслеживает доставку, получает и оценивает.**
  - **Персона + контекст:** Ана заказала пиццу по телефону, ждёт дома, приложение ставить не хочет.
  - **Entry state:** не авторизована, без приложения; получила сообщение в Viber (или SMS).
  - **Path:** приходит «Ваш заказ из {магазин} в пути, ориентировочно к 16:00» + ссылка → Ана открывает ссылку в браузере → видит магазин, статус «в пути», расчётное время → готовится принять.
  - **Climax:** курьер приезжает к нужному времени, Ана уже готова — быстрая передача, горячая пицца.
  - **Resolution:** через **ETA + 30 мин** Javi присылает Ане второе сообщение «как прошла доставка? оцените» со ссылкой → Ана ставит **оценку 1–5**. Оценку видит Милан.
  - **РЕШЕНО:** обязательной отметки о завершении нет (ни от Милана, ни от Аны) — система не зависит от события «доставлено». Запрос оценки триггерится по времени (ETA + буфер). Оценка запрашивается **активно** отдельным сообщением (+1 сообщение на заказ — учтено в стоимости и opt-out).

### 2.3 Non-Users (v1)
- Магазины на курьерских службах с доставкой 1–2 дня (другая модель ETA).
- _(Снято гибридом: магазины без системы заказов теперь поддержаны через ручной ввод.)_

## 3. Glossary
- **Магазин (Merchant)** — бизнес-пользователь Javi (напр. пиццерия). Ведёт заказы в своей системе; имеет в Javi профиль с **origin**.
- **Origin** — адрес магазина, точка отсчёта маршрута для расчёта ETA. Задаётся один раз в профиле.
- **Получатель (Recipient)** — конечный покупатель; пользуется бесплатно, без логина и без приложения.
- **Доставка (Delivery)** — единица работы Javi: один заказ → один получатель, с адресом назначения, телефоном, статусом и ETA. Поступает из системы магазина.
- **ETA** — расчётное время прибытия = момент старта + время в пути origin → адрес получателя (маршрутный сервис с трафиком).
- **«Доставка началась» (Dispatch event)** — действие магазина/курьера, триггер расчёта ETA и уведомления получателю.
- **Страница статуса** (ранее «трекинг-страница») — публичная страница доставки по непредсказуемой ссылке, без приложения и логина. В MVP показывает статус + ETA + оценку; живой трекинг движения курьера — Vision.
- **Оценка доставки** — 1–5 от получателя после доставки.

## 4. Features

### 4.1 Настройка магазина и профиль (origin)
**Description:** Милан один раз регистрируется и задаёт адрес магазина — **origin**, точку отсчёта для ETA. Realizes UJ-1.

- **FR-1.** Магазин регистрируется/входит под ролью «магазин» и видит только свои доставки.
  - Данные одного магазина недоступны другому.
- **FR-2.** Магазин задаёт origin в профиле; адрес геокодируется и сохраняется с координатами.
  - Без заданного origin расчёт ETA невозможен → система подсказывает задать адрес.

### 4.2 Приём доставки (гибрид)
**Description:** Доставка (полный адрес + телефон получателя) попадает в Javi двумя путями. Ведение заказов (каталог/оплата) — НЕ Javi. Realizes UJ-1.

- **FR-3.** Магазин/курьер может быстро создать доставку: полный адрес получателя + телефон (+ описание/№ опц.).
  - Адрес геокодируется; телефон нормализуется к +381.
- **FR-4.** Телефон получателя валидируется/нормализуется к E.164 (RS); невалидный — блок с примером формата; немобильный/иностранный номер — предупреждение без блокировки (пометка флагом риска).
- **FR-5.** Адрес получателя геокодируется; при неудаче — ручное уточнение/точка на карте или откат на ручной ETA.
- **FR-6.** _(later, `[NON-GOAL for MVP]`)_ Внешняя система магазина создаёт доставку через API. В MVP закладываем модель данных, совместимую с этим, но реализуем только ручной ввод.

### 4.3 Старт доставки и расчёт ETA
**Description:** Ключевой момент: «Доставка началась» → расчётное время прибытия. Realizes UJ-1.

- **FR-7.** Магазин или курьер отмечает у доставки **«Доставка началась»**.
  - Фиксируется время старта; запускается расчёт ETA.
- **FR-8.** Javi рассчитывает ETA = время старта + время в пути origin → адрес получателя (маршрутный сервис с трафиком), пояс Europe/Belgrade.
- **FR-9.** При недоступности маршрута/геокода — fallback на ручной ввод ETA; поток не блокируется.
- **FR-10.** ETA показывается получателю как **верхняя граница**: «ориентировочно к HH:MM» (не точный момент и не диапазон) — психологически «не позже», снижает риск ощущения опоздания.

### 4.4 Уведомление получателю и каналы
**Description:** Получатель узнаёт, что доставка началась и когда ждать. Realizes UJ-1, UJ-2.

- **FR-11.** При «доставка началась» Javi отправляет получателю сообщение «ваш заказ из {магазин} в пути, ориентировочно к {время}» + ссылка на страницу статуса.
- **FR-12.** Канал Viber-first; при недоставке Viber (fail/timeout) — авто-fallback на SMS с тем же текстом и ссылкой.
- **FR-13.** Идемпотентность: одно событие не дублируется по каналам и при повторных нажатиях (logical_message_id).
- **FR-14.** Javi фиксирует по сообщению канал, статус (queued/sent/delivered/read/failed) и время; где канал поддерживает — отметку «прочитано» (Viber).
- **FR-15.** Шаблон сообщений на сербском (латиница) по умолчанию; строго сервисный текст, без маркетинга.
- **FR-16.** Окно рассылки **08:00–22:00** (Europe/Belgrade): сообщения отправляются только в этом окне; событие вне окна (типично — запрос оценки по ETA+30 поздно вечером) откладывается до открытия окна. Уведомление «в пути» в окно попадает естественно (доставки днём/вечером).

### 4.5 Страница статуса доставки (+ оценка)
**Description:** Перепроверяемый статус по ссылке без приложения и место для оценки. В MVP это **страница статуса**, а не живой трекинг: ETA рассчитан один раз и не движется (живой GPS-ETA — Vision). Realizes UJ-2.

- **FR-17.** Публичная страница доставки по непредсказуемому токену, без приложения и логина.
- **FR-18.** Показывает: магазин, статус «в пути», ETA, город/район назначения. Без телефона и полного адреса. Без карты-слежения за курьером в MVP.
- **FR-19.** После доставки на этой же странице — блок оценки 1–5 (захват оценки — FR-22). Страница — единый дом для статуса и оценки.
- **FR-20.** Rate limit запросов к странице; срок жизни ссылки ограничен после доставки.

### 4.6 Запрос оценки и оценка доставки
**Description:** Качество доставки измеряется оценкой получателя. Запрос — активный, по времени (надёжного события завершения нет). Realizes UJ-2.

- **FR-21.** Через **ETA + 30 минут** Javi активно отправляет получателю сообщение «как прошла доставка? оцените» со ссылкой. Одно на доставку; подчиняется окну рассылки (FR-16) и отписке (FR-23).
- **FR-22.** Получатель ставит оценку 1–5 одним касанием, без логина; повтор не плодит дубли.
- **FR-23.** Получатель может отписаться (STOP в SMS / ссылка на странице). После отписки не-критичные сообщения (в т.ч. запрос оценки) на этот номер не шлются; магазин видит статус «получатель отписан». Уровень отписки (номер vs магазин) — см. Open Questions. `[MVP: минимально, но связно]`

### 4.7 Статусы и сигналы магазину
**Description:** Милан видит, что происходит с его доставками, и может починить сбой. Realizes UJ-1.

- **FR-24.** Магазин видит по каждой доставке: статус уведомления (доставлено/прочитано/не доставлено), ETA, оценку (если есть).
- **FR-25.** При недоставке уведомления магазин может исправить телефон и переотправить уведомление (явным подтверждаемым действием; идемпотентность по FR-13 сохраняется).
- **FR-26.** _(опц.)_ Магазин может вручную отметить «Доставлено» — необязательно; система от этого не зависит.

## 5. Non-Goals (Explicit)
- **Ведение заказов** (каталог, приём заказа, оплата) — остаётся в системе магазина; Javi хранит только доставку.
- **Маршрутизация/диспетчеризация/оборачиваемость** (оптимизация загрузки курьера, объединение заказов, мульти-стоп) → Vision.
- **Доставка на запланированную дату** (уведомление о дате + утреннее напоминание) → фаза 2.
- **Push получателю «доставлено»/«неуспешно»**, обновление ETA в пути, живой GPS-трекинг курьера.
- **Интеграции с курьерскими службами** (модель 1–2 дня), **COD-сумма**.
- **Локализация en/ru** (после sr).
- Javi **не становится** агрегатором/маркетплейсом — мы не забираем заказы и клиента у магазина.

## 6. Success Metrics
_Таргеты ниже — стартовые гипотезы Этапа 0, уточняются по первым данным._

**Главные:**
- **Удержание магазина** — **≥60%** магазинов продолжают создавать доставки на 4-й неделе после старта.
- **Готовность платить** — **≥3** из первых тест-магазинов готовы платить **≥€100/мес** (целевой ценник). _Амбициозно для микро-пиццерии → ключевой пункт валидации: не выше ли это потолка._
- **Оценка доставки 1–5** — средняя **≥4.3**, доля оценок 4–5 **≥80%**.

**Валидация спроса (Этап 0):** гейт ≥5 заявок от магазинов ICP с лендинга → строим MVP.

**Вторичные сигналы:**
- **Открываемость ссылки** получателями (доходит ли ценность до конечного пользователя).

**Counter-metrics (за чем следить, чтобы не навредить):**
- Доля отписок (opt-out) получателей — растёт ли от двух сообщений на заказ.
- Стоимость сообщений + маршрутных запросов на доставку (юнит-экономика).
- Точность ETA / жалобы на «обещали и не приехали».

## 7. Non-Functional Requirements
- **Производительность:** расчёт ETA и отправка уведомления — **< 5 секунд** после «доставка началась».
- **Надёжность доставки сообщений:** статусы доставки фиксируются; сообщение не считается доставленным без подтверждения провайдера.
- **Приватность/ПДн (Сербия):** минимизация данных на публичной странице (без телефона/полного адреса), непредсказуемый токен, согласие получателя на основании договора магазина; тело сообщений строго сервисное.
- **Локализация:** сербский (латиница) по умолчанию.
- **Мобайл-first:** вид магазина/курьера рассчитан на телефон «на бегу».
- **Безопасность:** изоляция данных магазинов; rate limit публичной страницы.

## 8. Dependencies & Risks
- **Маршрутный/картографический сервис** (геокодинг + ETA с трафиком по Сербии) — критическая зависимость; от него точность ETA и стоимость на заказ. _Риск: качество геокодинга вне крупных городов._
- **Провайдер Viber/SMS** — Viber Business + SMS с alphanumeric sender ID; нужны delivery/read receipts и обработка opt-out. _Риск: доступность/цена Viber Business для микро-объёмов; возможен старт на SMS._
- **Точность ETA** — завышенные обещания бьют по доверию; нужен буфер и формат-окно.
- **Принятие гибридного ввода** — ручной ввод = двойная работа для Милана; риск трения. _Митигейт: API позже; пока проверяем готовность терпеть ручной ввод._
- **Неподтверждённая готовность магазина платить** — главный риск Этапа 0; проверяется на лидах.

## 9. Open Questions
- Какой маршрутный сервис (Google Directions/Distance Matrix, Mapbox, ORS/OSRM)? Геокодинг сербских адресов, трафик, цена.
- Какой провайдер сообщений (Infobip и т.п.)? Коды ошибок, receipts, STOP/opt-out.
- Viber-first сразу или старт на SMS (зависит от доступности/цены Viber Business)?
- Opt-out на уровне номера или магазина; юр. оформление согласия (оферта/DPA), статус у RATEL/Poverenik.
- Приемлемо ли магазину двойное внесение адреса до появления API?
- Приемлемая стоимость на доставку при двух сообщениях (в пути + оценка) при ценнике €100/мес — сходится ли юнит-экономика?
- Срок хранения ПДн (телефоны/адреса/история уведомлений) после доставки.
