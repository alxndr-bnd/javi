## Вход 1 — Бриф

### Покрыто в PRD
- **Видение и позиционирование «анти-агрегатор»** — перенесено дословно (PRD §1 Vision, §5 Non-Goals: «Javi не становится агрегатором/маркетплейсом»). Тон «UX уровня Wolt/Glovo для своих курьеров без комиссии 30%» сохранён.
- **Beachhead** — пиццерии/кафе со своей доставкой день-в-день (PRD §1, §2).
- **JTBD магазина и получателя** — раскрыты в PRD §2.1 как персоны Милан/Ана.
- **Solution-флоу** (origin → адрес+телефон → «Доставка началась» → ETA → Viber/SMS + ссылка → оценка 1–5) — FR-1…FR-22.
- **Что отличает** (vs Wolt/Glovo, vs ручной Viber, vs курьерские службы 1–2 дня) — частично в Vision и Non-Goals (исключены курьерские службы 1–2 дня).
- **Success Criteria** — удержание магазина, готовность платить, оценка 1–5, гейт ≥5 заявок Этапа 0 — все перенесены (PRD §6).
- **Scope MVP / явно вне MVP** — в основном перенесён; Vision, фаза 2, COD, en/ru — корректно помечены как Non-Goals/later.

### Пробелы (потеряно или ослаблено)
1. **Связка «Доставлено» → запрос оценки потеряна / переосмыслена.** Бриф (Scope MVP): «Минимальная отметка “Доставлено” магазином (одно нажатие) — **закрывает доставку и включает запрос оценки**». В PRD отметка «Доставлено» сделана **опциональной** (FR-23, «система от этого не зависит»), а запрос оценки переведён на таймер ETA+30 мин (FR-19). Это сознательное решение PRD (зафиксировано в §2.2 «РЕШЕНО»), но оно **прямо расходится** с брифом — см. раздел «Расхождения».
2. **Вторичная/качественная метрика «открываемость трекинг-ссылки» опущена.** Бриф §Success Criteria: «открываемость трекинг-ссылки получателями» и субъективное «стало тише» по вопросам «где заказ?». В PRD §6 этих сигналов нет (есть только counter-metrics). Качественный сигнал ценности продукта молча выпал.
3. **Формулировка «кто платит» / денежный мотив магазина** в PRD ужата. Бриф детально разбирает разрыв «боль у получателя — деньги у магазина» (качество → повторные заказы, экономия на комиссии). В PRD это сжато до одной строки Vision; для приоритизации и валидации Этапа 0 этот тезис был важен.

### Расхождения с брифом
- **Отметка «Доставлено» и триггер оценки** (см. Пробел №1): бриф — обязательная MVP-отметка, закрывающая заказ и запускающая оценку; PRD — опциональная отметка + таймерный триггер оценки. Расхождение существенное, но в PRD объяснено и помечено как «РЕШЕНО».
- **Число сообщений на заказ.** Бриф описывает **одно** уведомление («в пути»), оценка собирается пассивно на трекинг-странице после отметки «Доставлено». PRD вводит **второе активное сообщение** «оцените» (FR-19) → 2 сообщения на заказ. PRD это учитывает в counter-metrics (доля opt-out, стоимость), но это отклонение от брифа.

### Пробелы (MVP use cases / FR без отражения в PRD)
1. **UC-quiet-hours (окно 09:00–20:00) — MVP в каталоге, но в PRD нет FR.** Каталог: UC-quiet-hours в списке «Приоритет MVP», FR-31/FR-32 (откладывание событий вне окна, отправка только последнего актуального состояния). В PRD §4 нет ни одного FR про окно рассылки. В Non-Functional/комплаенс тоже не упомянуто. Каталожная рекомендация смягчает («для день-в-день почти неактуально, но защититься стоит»), но формально MVP-UC и два FR молча выпали.
2. **Ручная переотправка уведомления и исправление номера (FR-18, FR-20 каталога) — потеряны.** Каталог-MVP: магазин может явно переотправить уведомление отдельным подтверждаемым действием (отличая от случайного дубля) и при недоставке исправить номер и переотправить. В PRD есть только просмотр статуса (FR-22), но нет FR на ручную переотправку / коррекцию номера. Это операционно важный путь восстановления при недоставке.
3. **Предупреждение для немобильного/иностранного номера (FR-4 каталога) ослаблено.** Каталог: невалидный → блок; немобильный/иностранный → **предупреждение без блокировки**. PRD FR-4 описывает только блокировку невалидного; мягкий предупредительный путь опущен.
4. **FR-14 каталога (имя магазина-отправителя в каждом уведомлении)** — в PRD присутствует в тексте шаблона FR-11 («из {магазин}»), но как самостоятельное требование не выделено. Покрыто де-факто, риск низкий.
5. **UC-dedupe-delivery / FR-37 (предупреждение о вероятном дубле доставки)** — в каталоге это `edge` + FR-37; в PRD отсутствует. Допустимо для edge-приоритета, фиксируем как сознательно не-MVP.

### Расхождения с каталогом
- **Формат ETA — прямое противоречие.** Каталог FR-10: ETA может показываться **как диапазон/окно** («~15:00–15:45»), что снижает риск ложных обещаний. PRD FR-10: ETA показывается **как верхняя граница** «ориентировочно к HH:MM» — и явно оговаривает «**не точный момент и не диапазон**». Две разные трактовки одного решения; PRD сознательно отверг окно в пользу верхней границы, но это противоречит букве каталога.
- **«Доставлено» как обязательное MVP-событие.** Каталог: UC-mark-delivered в MVP, FR-39 (одно действие, закрывает заказ). PRD: FR-23 «опц.», система не зависит. То же расхождение, что и с брифом.
- **Триггер запроса оценки.** Каталог связывает оценку с фактом «Доставлено» (FR-39 → FR-40, оценка на странице после доставки). PRD триггерит запрос оценки таймером ETA+30 мин (FR-19) независимо от отметки. Расхождение по механике.
- **Адрес/ETA — согласовано (искажения нет).** Каталог финально уточнил: адрес отправки = только origin для авто-ETA, получателю показывается максимум город. PRD это воспроизводит верно (FR-2 origin, FR-17 «город/район назначения, без полного адреса»). Открытая развилка каталога «курьерская служба vs своя доставка» закрыта в пользу своей доставки — PRD корректно вынес модель 1–2 дня в Non-Goals.

### Комплаенс / приватность — покрытие
- **Покрыто:** минимизация данных на публичной странице (FR-17, NFR §7), непредсказуемый токен (FR-16), rate limit + срок жизни ссылки (FR-19a), основание обработки ПДн = исполнение договора магазина (NFR §7), строго сервисный текст (FR-15), opt-out (FR-21). Открытые юр-вопросы (RATEL/Poverenik, уровень opt-out, DPA/оферта) перенесены в §9 Open Questions.
- **Пробел:** каталог FR-32 (при нескольких отложенных событиях слать только последнее актуальное) выпал вместе с темой quiet-hours.
- **Пробел:** срок хранения телефонов/адресов/истории уведомлений после доставки (открытый вопрос каталога §Комплаенс) в PRD не зафиксирован даже как Open Question — только срок жизни ссылки. Retention данных ПДн потерян.
