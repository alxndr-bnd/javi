# Use Cases: Javi

Javi — сервис автоматических уведомлений о доставке для небольших магазинов в Сербии. Цель документа — зафиксировать, что именно система должна делать на Этапе валидации спроса и первого MVP: какие роли участвуют, как проходит самый простой сценарий и какие требования из него вытекают. Продукт закрывает информационный разрыв между моментом «магазин отдал посылку курьеру» и моментом «покупатель её получил»: вместо ручной переписки в Viber магазин один раз заводит отправку, а покупатель получает уведомление и ссылку трекинга без установки приложения.

Якорный кейс (MVP) — **доставка день-в-день своим курьером**: магазин отгружает заказ, отмечает «доставка началась», система рассчитывает время прибытия (ETA) по маршруту от адреса магазина до адреса получателя и сообщает покупателю «в пути, прибудет ~{время}» + трекинг-ссылка. Это модель Wolt/Glovo для собственных доставок магазина. Фаза 2 — доставка на запланированную дату (уведомление о дате → утреннее напоминание → далее тот же финальный отрезок «старт → ETA»).

## Акторы

| Актор | Описание |
|---|---|
| Магазин (владелец/сотрудник, отправитель) | Главный платящий пользователь. Малый продавец в Сербии (Instagram-/Facebook-шоп, домашнее производство, нишевый e-commerce). Создаёт отправку, вводит адреса, телефон и язык получателя, вручную меняет статус доставки и ETA. Его действия порождают события уведомлений. На старте владелец и сотрудник — одно лицо. |
| Получатель (покупатель) | Конечный благополучатель, для него сервис бесплатен. Получает уведомления в Viber (SMS как fallback), открывает публичную трекинг-страницу по одной ссылке без установки приложения и без логина. Может отписаться. Часто платит при получении (COD/pouzeće). |
| Курьер / курьерская служба | Внешний оператор доставки (Pošta/Post Express, BEX, AKS, D Express, City Express и др.), который физически везёт посылку. В MVP НЕ интегрирован и НЕ является пользователем системы — статусы и ETA магазин ставит вручную. Источник реального события «отправлено / в пути / доставлено». Полноценным актором становится только в Vision. |

Соблюдает окно рассылки 09:00–20:00, opt-out и строго сервисное содержание. |
| Провайдер Viber / SMS | Внешний шлюз доставки сообщений (Viber Business + SMS-агрегатор с alphanumeric sender ID). Доставляет сообщение, возвращает статус (delivered/failed), инициирует fallback. |
| Администратор платформы | Оператор сервиса Javi (на старте — сам владелец продукта). Онбордит магазины, настраивает отправителей Viber/SMS, следит за доставляемостью и расходом. В простейшем кейсе не участвует; роль вспомогательная. |
| Внешний сервис геокодинга / маршрутизации | Внешний провайдер карт/маршрутов для перевода адреса в координаты и оценки времени в пути. Нужен только для автоматического ETA — это post-MVP. |

> **БОЛЬШОЕ уточнение (заказчик, 2026-06-01).** Первая боль для решения — **доставка день-в-день своим курьером с расчётным временем прибытия** (модель Wolt/Glovo для собственных доставок магазина), а НЕ курьерская служба на 1–2 дня. Это меняет якорный кейс: адрес отправки нужен (origin маршрута), адрес получателя нужен полный (для расчёта маршрута), ETA — это рассчитываемое **время сегодня**, триггер — событие «доставка началась». Авто-расчёт ETA переходит в MVP. Доставка на запланированную дату — фаза 2 (см. ниже).

## Якорный use case (MVP): доставка день-в-день с расчётным ETA

**ID:** UC-same-day-delivery
**Актор:** Магазин (отправитель) + Система Javi
**Триггер:** Магазин отгружает заказ в доставку (свой курьер выезжает) и отмечает «Доставка началась».

**Предусловия:**
- Магазин зарегистрирован; в профиле задан **адрес магазина (origin)** — точка отсчёта маршрута/ETA.
- Есть полный адрес получателя и его мобильный телефон.

**Основной поток:**
1. Магазин создаёт заказ: полный адрес получателя (улица + дом), имя, телефон; описание/№ заказа (опц.). Origin берётся из профиля магазина — повторно вводить не нужно.
2. Система валидирует/нормализует телефон (+381, E.164) и геокодирует адрес получателя.
3. В момент отгрузки магазин нажимает **«Доставка началась»**.
4. Система рассчитывает ETA = текущее время + время в пути origin→destination (маршрутный сервис с учётом трафика).
5. Система генерирует трекинг-страницу (непредсказуемый токен) и отправляет получателю уведомление: «Ваш заказ из {магазин} в пути, ориентировочно прибудет ~{время}» + ссылка (Viber, SMS fallback).
6. Получатель открывает ссылку без приложения: видит магазин, статус «в пути» и расчётное время.
7. (later) По завершении магазин отмечает «Доставлено» → опциональное финальное уведомление.

**Альтернативные / исключительные потоки:**
- *Адрес получателя не геокодируется* → магазин уточняет адрес или ставит точку на карте; если не вышло — fallback на ручной ввод времени.
- *Маршрутный сервис недоступен* → fallback на ручной ввод ETA магазином, без блокировки потока.
- *Нет Viber / Viber не доставлен* → автоматический fallback на SMS с тем же текстом и ссылкой (UC-channel-fallback).
- *Невалидный телефон* → сохранение блокируется с примером формата.
- *Повторный клик «Доставка началась» / ретрай* → дубль подавляется по logical_message_id.
- *Сильное отклонение от ETA в пути* → (later) уведомление-обновление времени.

**Нужные данные:**
- Origin: адрес магазина (из профиля) + его координаты
- Полный адрес получателя (улица + дом) + координаты (геокод)
- Имя получателя
- Телефон получателя (E.164, +381)
- Описание / № заказа (опц.)
- Время отгрузки (старт доставки)
- Расчётный ETA (время прибытия) + способ расчёта (авто/ручной)
- Статус (создан / в пути / доставлен)
- Публичный неугадываемый токен трекинг-ссылки
- Канал и статус доставки уведомления (queued/delivered/failed) + время

**События уведомлений:**
- «Доставка началась» → уведомление получателю с расчётным ETA + трекинг-ссылка (ключевое и единственное обязательное событие MVP).
- (later) Значимое изменение ETA → уведомление-обновление.
- (later) «Доставлено» → финальное подтверждение.

**Постусловия:**
- Создан заказ с активной трекинг-ссылкой и зафиксированным расчётным ETA.
- Получателю доставлено (или поставлено в очередь) уведомление с временем прибытия; зафиксированы канал, статус и время.
- Получатель может в любой момент открыть трекинг-страницу с актуальным статусом/ETA.

## Use case фазы 2: доставка на запланированную дату

**ID:** UC-scheduled-delivery
**Актор:** Магазин (отправитель) + Система Javi
**Триггер:** Магазин оформляет заказ с доставкой на будущую дату (не сегодня).

**Основной поток:**
1. Магазин создаёт заказ и указывает **плановую дату доставки** (+ опц. часть дня/слот).
2. Система уведомляет получателя: «Доставка вашего заказа из {магазин} запланирована на {дата}» + трекинг-ссылка.
3. **Утром дня доставки** система отправляет напоминание: «Сегодня доставка вашего заказа из {магазин}».
4. Когда в этот день магазин нажимает «Доставка началась» → запускается поток **UC-same-day-delivery** (расчёт ETA-времени и уведомление «в пути ~{время}»).

**Дополнительные данные:** плановая дата (+ слот); флаги «дата сообщена», «утреннее напоминание отправлено».

**События уведомлений:** создание → уведомление о плановой дате; утро дня X → напоминание; старт доставки → ETA (как в MVP).

**Зависимость:** фаза 2 надстраивается над MVP — финальный отрезок (старт → ETA → прибытие) идентичен якорному кейсу.

## Функциональные требования

### Данные доставки и адреса
- **FR-1.** Магазин может создать заказ (один заказ — один получатель) с полями: полный адрес получателя, имя и телефон получателя, описание/номер заказа (опц.). Адрес отправки (origin) в заказ не вводится — берётся из профиля магазина.
- **FR-1b.** Магазин задаёт в профиле адрес магазина (origin); он используется как точка отсчёта маршрута/ETA для всех заказов день-в-день.
- **FR-2.** Для доставки день-в-день адрес получателя нужен **полный** (город + улица + дом) — он геокодируется для расчёта маршрута. (Для относительного/ручного ETA достаточно города.)
- **FR-3.** Телефон получателя валидируется и нормализуется к E.164 для региона Сербия (RS): +381 / 00381 / ведущий 0 приводятся к каноническому виду.
- **FR-4.** При невалидном телефоне сохранение блокируется с примером корректного формата; при немобильном/иностранном номере — предупреждение без блокировки.
- **FR-5.** Система геокодирует адрес получателя; при неудаче — позволяет уточнить адрес/поставить точку на карте или откатиться на ручной ETA.
- **FR-6.** На публичной трекинг-странице из адреса получателя показывается минимум (город/район), без полного адреса и без телефона.

### ETA (расчёт времени прибытия)
- **FR-7.** В момент события «доставка началась» система **автоматически рассчитывает ETA** = текущее время + время в пути origin (магазин) → destination (получатель) по маршрутному сервису с учётом трафика.
- **FR-8.** ETA-текст формируется человекочитаемо с учётом часового пояса Europe/Belgrade и языка шаблона (sr/en/ru); ETA день-в-день — это время («~15:30»), а не дата.
- **FR-9.** При недоступности маршрутного сервиса или неудачном геокодинге система откатывается на ручной ввод ETA магазином — поток не блокируется.
- **FR-10.** ETA показывается получателю как верхняя граница «ориентировочно к HH:MM» (не точный момент и не диапазон) — решено в PRD. _(Снижает риск ложных обещаний.)_
- **FR-11.** (later) При значимом отклонении фактического хода доставки от ETA система обновляет время и шлёт уведомление-обновление; уведомление — только при реальном изменении, не при пересчёте того же значения.
- **FR-12.** (later) Поддержка нескольких остановок (маршрут из нескольких доставок) с per-stop ETA — расширение якорного кейса «один заказ».

### Уведомления и каналы
- **FR-13.** Система автоматически порождает уведомление получателю при смене статуса на «выехала / в пути».
- **FR-14.** Каждое уведомление содержит актуальную трекинг-ссылку и указывает имя магазина-отправителя.
- **FR-15.** Уведомления отправляются по принципу Viber-first: первая попытка — всегда Viber.
- **FR-16.** При неподтверждённой доставке через Viber (failed/undelivered или таймаут) система автоматически переотправляет то же сообщение по SMS.
- **FR-17.** Система не дублирует одно событие по обоим каналам при успешной доставке через Viber (идемпотентность по logical_message_id; защита от двойных кликов и ретраев).
- **FR-18.** Магазин может явно переотправить уведомление отдельным подтверждаемым действием, отличая это от случайного дубля.
- **FR-19.** Система фиксирует по каждому уведомлению канал, статус (queued/sent/delivered/failed) и время; сообщение не считается доставленным без подтверждения delivered.
- **FR-20.** Магазин видит статус доставки уведомления и при недоставке может исправить номер и переотправить.
- **FR-21.** SMS отправляются с зарегистрированным alphanumeric sender ID (MTS/Yettel/A1).
- **FR-22.** (later) Система поддерживает события «доставлена», «не доставлена», «перенос» как источники уведомлений, не требуя их для простейшего кейса.

### Содержание и язык
- **FR-23.** Содержание сообщений формируется из шаблонов на языке заказа: сербский (латиница) по умолчанию, плюс английский и русский.
- **FR-24.** Сообщения строго сервисные (статус + ETA + ссылка), без маркетинга.
- **FR-25.** Трекинг-страница поддерживает язык получателя (sr/en/ru), сербский по умолчанию.

### Трекинг и приватность
- **FR-26.** Система генерирует публичную трекинг-страницу, доступную по одной ссылке без установки приложения и без логина.
- **FR-27.** Трекинг-ссылка содержит длинный непредсказуемый токен (не последовательный ID) и не раскрывает данные других заказов.
- **FR-28.** Публичная страница показывает только безопасный минимум: магазин, статус, ETA, город назначения; без телефона и полного адреса.
- **FR-29.** Система ограничивает частоту запросов к трекинг-странице и задаёт срок жизни ссылки после доставки/отмены.
- **FR-30.** (later) При отмене отправки трекинг-ссылка деактивируется, запланированные уведомления отменяются, опционально отправляется уведомление об отмене уже извещённому получателю.

### Часы рассылки, согласие и комплаенс
- **FR-31.** Уведомления отправляются только в окне 09:00–20:00 по местному времени; события вне окна откладываются и уходят при его открытии.
- **FR-32.** При нескольких отложенных событиях по одному заказу отправляется только последнее актуальное состояние.
- **FR-33.** Персональные данные (телефон, адрес получателя) обрабатываются на основании исполнения договора магазина с покупателем; тело уведомления строго сервисное.
- **FR-34.** (later) Каждое SMS содержит инструкцию отписки (opt-out); система обрабатывает отписку по STOP и по ссылке/кнопке на трекинг-странице.
- **FR-35.** (later) После отписки система не отправляет дальнейшие уведомления на этот номер и показывает магазину статус «получатель отписан».

### Оценка доставки
- **FR-39.** Магазин может отметить заказ «Доставлено» одним действием (статус, без отдельного push-уведомления получателю в MVP).
- **FR-40.** После доставки трекинг-страница предлагает получателю оценить доставку по шкале 1–5 (одно касание, без логина); повторная оценка не плодит дублей.
- **FR-41.** Магазин видит оценки своих доставок (средняя и распределение) как сигнал качества.

### Учётные записи
- **FR-36.** Магазин может зарегистрироваться/войти под ролью «магазин» и видеть только свои заказы.

### Защита от ошибок (edge)
- **FR-37.** Система предупреждает о вероятном дубле доставки при совпадении телефона получателя и даты в недавнем окне.

### Администрирование (later)
- **FR-38.** Администратор настраивает отправителей Viber Business и SMS sender ID и контролирует доставляемость/расход.

## Открытые вопросы

**Гипотезы заказчика (проверить в первую очередь):**
- **Два адреса:** УТОЧНЕНО (заказчик, 2026-06-01) — адрес отправки получателю НЕ показывается; он нужен только как **origin (точка отсчёта) для расчёта ETA**. Следствие: необходимость адреса отправки полностью завязана на способ расчёта ETA (см. ниже). При ручном ETA адрес отправки не нужен вообще; нужен только при авто-ETA по маршруту. Получателю в любом случае показывается максимум **город** назначения.
- **ETA — зависит от способа доставки (ключевая развилка):**
  - *Курьерская служба* (Pošta/BEX/D Express/… — типичная модель COD в Сербии): фактический ETA определяется **расписанием/SLA курьера** (обычно 1–2 дня), а НЕ географическим расстоянием. Фургон едет по маршруту, а не «из точки А в точку Б». → авто-расчёт по origin→destination здесь вводит в заблуждение; оба адреса для ETA не нужны, достаточно «1–2 рабочих дня» или ручной даты.
  - *Своя доставка / тот же город* (магазин везёт сам): ETA можно оценить по расстоянию/маршруту → тогда адрес отправки осмыслен как origin.
  - Открытый вопрос формата: относительный («завтра / 1–2 дня») vs дата vs интервал/слот (3 ч, как у GLS).
- **Канал:** Viber-first или старт на SMS? Доступность и стоимость Viber Business для микро-объёмов без минимальных контрактов под вопросом — возможно, MVP стартует на SMS, а Viber добавляется позже.

**Маршрут и ETA (день-в-день) — критично для MVP:**
- Какой маршрутный/картографический сервис: Google Maps (Directions/Distance Matrix), Mapbox, ORS/OSRM? Нужен трафик + покрытие адресов Сербии. От него зависят точность ETA и стоимость на заказ.
- Качество геокодинга сербских адресов (особенно вне Белграда/Нови-Сада): хватит ли провайдера или нужна ручная установка точки на карте?
- Учитывать ли время на саму передачу/подъезд (буфер), чтобы ETA не был систематически оптимистичным?
- Пересчитывать ли ETA в пути (трекинг живого положения курьера) или достаточно одной оценки в момент старта? (живой трекинг — later, нужен источник геопозиции курьера.)

**Каналы и провайдер:**
- Какой именно провайдер (Infobip и т.п.)? От него зависят коды ошибок, STOP-обработка и delivery receipts (FR-16/FR-19/FR-20).
- Таймаут ожидания Viber-доставки перед SMS fallback, чтобы не словить дубль при поздней доставке.
- Лимит символов и стоимость сегментов для длинных SMS (кириллица vs латиница), как обрезать, сохранив ссылку и opt-out.
- Кириллица или латиница в сообщениях, и кто задаёт язык — магазин или авто.

**Уведомления и поведение:**
- Какие смены статуса триггерят уведомление по умолчанию (все четыре или только «выехал» + «доставлен»), чтобы не спамить.
- Нужны ли магазину тумблеры «не уведомлять о доставке/неудаче» уже в MVP.
- Анти-спам лимит: сколько уведомлений-обновлений (ETA/перенос) по одному заказу за сутки.
- Какой отправитель указывается в Viber — бренд Javi или имя магазина.
- Откуда берётся фактический статус: ручной ввод магазином (приемлемое ли это трение?) или интеграция с курьерами.

**Комплаенс и приватность:**
- Действует ли opt-out на уровне номера телефона глобально или на уровне заказа/магазина (юридически безопаснее — номер, но это бьёт по другим магазинам).
- Считается ли «доставка сегодня» исключением из opt-out, или после STOP не слать ничего.
- Кто отвечает за согласие получателя — магазин гарантирует основание, Javi исполняет; нужна формулировка в оферте/DPA.
- Фиксированный сербский часовой пояс (CET) для окна 09:00–20:00 или по ТЗ получателя.
- Подтвердить у RATEL/Poverenik, попадает ли уведомление под почтовую услугу и сколько данных можно показывать на публичной странице.
- Срок хранения телефонов/адресов и истории уведомлений после доставки.

**Учётные записи и модель данных:**
- Один пользователь на магазин или несколько (владелец + сотрудник) в MVP?
- Переиспользуем существующую `CustomUser.role` (customer/carrier) или вводим новую роль «shop»?
- Текущая Django-модель (Request/Response/CustomUser) описывает биржу перевозок, а не уведомления — её нужно переосмыслить под «Отправка + Уведомление + ТрекингСсылка».
- Принимаем ли иностранные (не +381) номера; откуда брать справочник сербских мобильных префиксов (достаточно ли libphonenumber).
- Нужно ли хранить трек-номер курьера уже в MVP для будущих интеграций.

**Рынок:**
- Реальная частота COD среди целевых микро-магазинов (>50% или ниша) — определяет приоритет UC-cod-amount-reminder.
- Не дублирует ли Javi бесплатные SMS-уведомления, которые курьеры уже дают в своих кабинетах.

## Рекомендация по объёму MVP (Этап 1)

- **Взять ядро якорного кейса (день-в-день):** регистрация магазина + адрес магазина (origin) в профиле; создание заказа (полный адрес получателя + телефон); геокодинг адреса; кнопка «Доставка началась»; авто-расчёт ETA по маршруту origin→destination; одно уведомление «в пути, прибудет ~{время}» + трекинг-страница с непредсказуемым токеном.
- **Маршрут/ETA — ключевая зависимость MVP (не откладывать):** нужен маршрутный сервис с трафиком (геокодинг + routing). Обязателен fallback на ручной ETA при сбое, чтобы поток не блокировался. Это главное техническое решение этапа — выбрать провайдера карт/маршрутов (вопрос в Открытых).
- **Один канал на старте, fallback заложить архитектурно:** Viber-first с SMS fallback; если Viber Business дорог/долог — стартуем на SMS, Viber следующим шагом. Обязательно идемпотентность (logical_message_id) и фиксация статуса доставки.
- **Минимальный комплаенс с первого дня:** строго сервисный текст, непредсказуемый токен и минимизация данных на публичной странице, opt-out в SMS. (Окно 09:00–20:00 для день-в-день почти неактуально — доставки и так днём, но защититься стоит.)
- **Сербский (латиница) по умолчанию** для сообщений и страницы; en/ru — следующим шагом.
- **Отложить (фаза 2+):** доставку на дату (уведомление о дате + утреннее напоминание), уведомления «доставлено»/«неуспешно», обновление ETA в пути, мульти-стоп маршрут, отмену, COD-сумму, админ-настройку отправителей, любые интеграции с курьерскими службами (модель 1–2 дня). Подключаются после подтверждения, что базовый кейс «в пути + расчётное время» снижает неуспешные доставки.
